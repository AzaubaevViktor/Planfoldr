# Phase 3: Динамическая оркестрация с ролями и живым деревом тикетов

## Какие проблемы решаем
Работа языковых моделей недетерминирована
Языковая модель может забывать инструкции или срезать углы при большом контексте
Для большинства задач не нужны большие модели с дорогими токенами/подписками, часто достаточно дешёвых/локальных моделей
Языковая модель не может гарантировать, что верификация, написанная в инструкции, будет выполняться всегда
Поставщики языковой модели могут ухудшить качество работы после дообучения и инструкции, которые работали раньше, перестанут работать
Языковая модель лучше выполняет небольшие ограниченные действия

## Суть

Phase 2 доказала главный принцип: модель — не центр управления. Детерминированный runtime управляет флоу, модель — исполнитель внутри задач.

Phase 3 делает следующий шаг: runtime становится субстратом для динамической оркестрации, где исполнение порождает новые исполнения. Тикет — это не запись в базе. Тикет — это цикл. Цикл создаёт тикеты. Тикеты становятся циклами. Всё это наблюдаемо в реальном времени.

## Что не так с Phase 2

1. **Статический граф задач** — весь флоу описан заранее в YAML. В реальной работе задачи появляются по ходу.
2. **Нет ролей** — все модели одинаковы. Нет специализации, только промптами
3. **Нет очередей** — модели не забирают работу самостоятельно, они запускаются поштучно.
4. **Вложенные циклы не порождаются динамически** — только статически описанные `nested_cycles`.
5. **Visibility** — отчёт строится после завершения. Live view ограниченный и неудобный.

## Базовый цикл

Базовый цикл — атомарная единица работы над тикетом. Его грамматика:

### Context explorarion
У модели есть доступ к локальной памяти, которая исчезнет после завершения работы над тикетом.
У модели есть доступ к тикетам и базе знаний.
Есть возможность редактировать ту часть тикетов и часть базы знаний, к которым он имеет доступ
Есть возможность создать тикет на одну из ролей, по шаблону
Есть возможность писать комментарии в тикете, призывать другие роли, даже ещё несуществующие (перенаправляются к создателю ролей, он может одобрить или отказать создание роли, появляется у него во входящих)
Есть возможность отвечать на комментарии в тикетах, куда его призвали
Есть возможность следить за вложенными циклами и изменениями
Есть список «входящих изменений» от циклов, тикетов, призывов итд

### Изменения в рабочей копии
У модели есть доступ к локальной памяти, базе знаний, текущему тикету, над которым идёт работа, и другим тикетам. 
Менять тикеты и оставлять комментарии модель в рамках изменений в рабочей копии не может.
Модель может использовать тулы, разрешённые в тикете.

### Command Verification
Запуск тестов / команд, сбор механических доказательств
Могут создаваться пользователем, могут меняться, удаление и добавление должно отражаться в аудит-логе


### Model Verification
Модель оценивает evidence против критериев тикета
На каждый evidence — отдельная задача


### Общие мысли по поводу базового цикла
Не каждый тикет требует всех четырёх фаз. Тикет типа `research` может состоять только из 1 и 4. Тикет типа `verify` — только из 3 и 4. Но последовательность не нарушается: нельзя верифицировать то, что ещё не изменено.

**Базовый цикл — это выполнение тикета.**

## Очередь и роль

Очереди разделены по направлениям

У очереди есть описание, в каких случаях и как нужно в эту очередь обращаться

За каждой очередью закреплены несколько ролей, которая задаётся prompt-ом

Есть управляющий очередью, который выбирает следующую задачу и может менять шаблон очереди (как создавать тикеты, json-ка). Управляющий переводит тикеты из статуса `("incoming")` в статус `("in_queue", priority)`

То есть: есть цикл и над тикетом, и над очередью.
Одно отличается от другого тем, что у цикла над тикетом цель это **достичь целей тикета**, а цикл над очередью — это выбрать и выполнить те задания над очередью, которые соответствуют цели очереди и всего проекта, а всё неважное откинуть и не выполнять.

Есть копатель, который непосредственно делает какой-то тикет

Роли может создавать особая роль «создатель ролей», на которую нужно сделать тикет и она сделает очередь и роли

## Выбор модели
Не все модели одинаково справляются с одной и той же задачей, поэтому нужно будет вести несколько статистик:
- Какая модель лучше справляется в целом
- Какая модель лучше справляется внутри роли
- Если модель два-три раза не справилась с тикетам, брать другую или выбирать более мощную из доступных
- Всё это хранить в аудите

По умолчанию мы считаем, что чем больше параметров в модели, тем она лучше (базовый счёт), но это может меняться как для всего процесса, так и для конкретных частей процесса

## Управление ресурсами и бюджетами
Поначалу будет использоваться одна модель из ollama
Далее будет возможность подключать модели по API (openai, antropic, ...)
После будет возможность использовать подписочные модели (codex, claude, ...)
Команды сначала просто запускаются, но после будут запускаться в ограниченном окружении

Бюджеты обновляются во время работы и считаются относительно:
- Для команд:
  - потребление cpu / ram
  - количество запусков команд
- Для правок файлов:
  - Количество изменений в файлах
  - (?) Количество добавленных / удалённых строк
- Для моделей:
  - Количество токенов
  - Количество потраченных денег
  - Количество запросов к модели
  - Количество GPU * RAM * часы для локальных моделей
- Общие
  - Количество созданных очередей, ролей и тикетов внутри каждой очереди и внутри всего проекта


При превышении бюджета даём закончить все ongoing-таски внутри циклов и не запускаем следующие таски, возвращаем из сценария `budget_exceeded`

### ollama
ollama предаставляет в ответе только количество токенов, а gpu * ram * время нужно будет считать для каждой модели исходя из `ollama ps`. Нужно также правильно считать если два тикета используют одну модель

### API
Там всё возвращается, и деньги, и количество токенов

### Запуск через подписку (нативные агенты)
Надо поисследовать, что там можно считать

## Тикет как единица исполнения

В Phase 2 тикет — это структура данных. В Phase 3 тикет — это описание одного запуска базового цикла.

Когда тикет переходит в `running`, runtime создаёт для него цикл. Когда цикл завершается — тикет получает статус (`done` / `failed`) и evidence.

Поля тикета:
- id тикета: `<роль или компонент>-номер`
- Название тикета
- Статус тикета
- Цель тикета
- Набор проверок для того, что цель тикета достигнута. Обязательно должны быть пройдены
- Предыдущие результаты проверок
- Доступы тикетаё
- Бюджет тикета
- Зависимости тикета
- Доступные тулы (должна согласовать особая роль «согласователь тулов»)
- Доступ к контексту
- Доступы к тулам
- метаинформация (кто, когда создал, история изменений, etc)
- Комментарии
  - От кого
  - Когда
  - Кого конкретно призывают
  - Текст комментария
  - Что нужно сериализованно


### Тикет failed
Если тикет не смог выполниться через несколько попыток (задаётся отдельно), то модели назначается штраф (в зависимости от сложности задачи, чем проще задачи - тем больше штраф) и тикет передаётся следующей модели.

## Динамическое создание тикетов

Исполняя тикет, модель может создавать новые тикеты. Это — главное отличие от Phase 2.

Создание тикета — явное действие через `create_ticket` tool. Не скрытое в тексте ответа, не угадываемое из контекста — явный вызов инструмента.

Примеры:

- **Разработчик** делает авторизацию → создаёт тикет `security-review` для роли `security`
- **Планировщик** получает задачу → разбивает на N тикетов для разных ролей
- **Исполнитель** обнаруживает нехватку контекста → создаёт тикет `research` для роли `researcher`
- **Верификатор** находит дыру → создаёт тикет `fix` для роли `developer`
- **Security** находит критическую уязвимость → создаёт `block` на тикет релиза

Каждое создание тикета — trace event с:
- кто создал (роль, цикл, execution id)
- почему (reason, evidence из текущего исполнения)
- связь с создателем (`spawned_by`)

Новые тикеты сразу попадают в очередь роли, должны быть отсмотрены управляющим очередью и либо:
- Переводятся из `incoming` в `in_queue, priority`
- Переводятся из `incoming` в `declined, cause` и видны только управляющему очередью

## Флоу работы

Цикл создаётся из формулировки задачи пользователем
Внутри самого верхнего цикла мы можем запросить дополнительную информацию у пользователя, призвав специально выделенную роль `@human`
Задачи верхнего цикла:
Context:
- Сформулировать общую цель проекта (будь это исправить тест или «сделай сервис который заработает мне 1M$/месяц, Make no mistak»)
- Сформулировать ограничения
- Понять, какие будут роли и тикеты, заполнить их
- Сформулировать вехи, задать для каждой из них например цель «сделать MVP, цель которого ...»
Execution:
- Выделить бюджет
- Отдать на выполнение другим циклам (работает автоматически исходя из нарезанных тикетов)
- Дождаться завершения либо исчерпания бюджета
Verification
- Автоматизированные проверки (начинсляют баллы за конкретную роль)
Model Verification
- Модель проверяет соответствие полученного результата требованиям и ограничениям

## Баллы моделям

Баллы начисляются или вычитаются каждой модели за (роль, сложность, очередь, тикет) и используются для выбора модели в следующих попытках.

**Критерии начисления (+):**
- ✅ Тикет выполнен без ошибок → положительный небольшой балл за корректный вывод
- ✅ Tикet верифицирован (пройден Model Verification + Command Verification) → весовые баллы модели, которая делала тикет
- ✅ Экономия бюджета по сравнению с выданными баллами → бонусные баллы относительно разницы модели
- ✅ Быстрое достижение цели → временной бонус за эффективность
- ✅ Тикет прошёл проверку → бонус за правильный тикет

**Критерии вычитания (-):**
- ❌ Tикет failed после нескольких попыток → штраф (чем проще задача, тем больше штраф)
- ❌ Бюджет исчерпан в рамках тикета → тяжёлый штраф по размеру израсходованного бюджета
- ❌ Неэффективное использование токенов (>80% от max_tokens_per_ticket) → penalty за waste
- ❌ Model Verification не прошёл, но модель пыталась верифицировать без достаточных proofs → штраф за ложную уверенность
- ❌ Создание излишних child ticket'ов без обоснованного evidence → штраф за over-engineering после отклонения тикета

**Перебор моделей:**
- Ведётся статистика: какая модель лучше справляется в целом, внутри роли, для типов задач
- Если модель 2-3 раза не справилась с тикетом одного типа, берём другую или более мощную из доступных
- По умолчанию считаем, что чем больше параметров модели, тем она лучше (базовый счёт), но это может меняться по конкретным метрикам для каждой роли

## Циклы
Правила вложенности циклов:

- `constraints` идут вниз от родителя к потомку
- `facts` (результаты, evidence) идут вверх через явно описанный output
- Parent cycle владеет целостностью дерева; child не может объявить весь проект завершённым
- Бюджет делегируется явно при порождении child цикла
- Parent может создавать множество подциклов, и следить за изменениями в их состояниях на этапе работы с контекстом


## Ролевая система

У каждой роли есть:
- id
- промпт, описывающий специализацию языковой модели + промпт от очереди
- скоуп доступа к тулам + скоуп доступа от очереди

Каждая роль по сути это специализация, но из-за того, что роль может быть закреплена за очередью, это еще и специализация над очередью. 
То есть у нас будет одна роль управляющего над тикетами, но для каждой очереди туда будут подмешиваться дополнительные моменты от каждой очереди. 

## Очереди

У очереди есть название. Внутри нее лежат тикеты в разных статусах. 

За очередью закреплено несколько ролей:
- Одна роль — это роль, которая управляет самой очередью, приоритезирует, обрабатывает входящий поток, «управляющий тикетами», его 
- И вторая роль это непосредственно копатель, который выполняет задачи, в зависимости от того, какие задачи принес управляющий этикетами. 

Когда тикет переходит в `ready` (зависимости выполнены), он появляется в очереди своей роли. Модель с ролью Исполнитель забирает тикет и начинает цикл.

Одна роль может обслуживать несколько очередей. Одну очередь могут обслуживать несколько экземпляров роли параллельно (для параллельной обработки независимых тикетов).

Примеры очередей для сценария "разработка модуля":

```
developer queue     → code, tests, refactor
security queue      → security_review, audit
research queue      → research, documentation
verification queue  → verify, benchmark
orchestration queue → plan, decompose, review_tree
```

## Создатель ролей

Есть роль, которая создаёт другие роли.

Создатель ролей (`birthgiver`) умеет:
- Анализировать описание задачи и текущее состояние дерева
- Определять, каких ролей не хватает
- Создавать новые роли с нужными параметрами (`create_role` tool)
- Открывать новые очереди
- Подмешивать к очередям роли

Основная задача создателя ролей, если он видит, что кому-то нужна конкретная роль:
- Если эта роль существует, ничего не делать, связать требуемой роль с уже существующей
- Если создатель роля считает, что эта роль не нужна и она излишняя, то обосновать отказ
- Если создатель роли видит, что роль действительно нужна, то создать очередь и набор ролей внутри этой очереди, которые будут эту очередь обслуживать. 

Пример: задача нестандартная и требует роли `performance_analyst`. Создатель ролей создаёт на ходу очередь с нужными ролями, toolset и budget scope.

## Visibility как первый класс

Visibility — это не опция, это основное свойство системы.

Visibility состоит из двух циклов.

### Потоковая работа
Каждое событие, каждое действие помечается в html логе. 
Выглядеть это должно как в выводе кодовых агентов. 
Взглянув в отлог, человек должен понимать, в рамках чего какая модель, с какими входными данными сделала что
Также должен быть кат с всей необходимой дополнительной информацией, по необходимости скатами внутри. 
Там должны быть входные данные, доступный контекст, доступные тулы. Это вход в какой-то execution.
Во время процесса работы должно быть понятно, что какая команда возвращается, что она вернула. Для модели должно быть понятно что она потоково возвращает где у нее thinking где она что-то выводит где использует тулы 
Так, например, для тулов может быть кат с коротким текстом крейтин тикет, а внутри информация о том, какой тикет по итогу создался.  

### Общее состояние
На второй штемой страничке должно быть динамически видно, какие модели работают, какие команды сейчас запущены, в рамках каких очередей, каких тикетов какая работа происходит. 
То есть должна быть возможность внутри определенной очереди открыть определенный тикет, посмотреть, что за модель там, что делает, какой у нее сейчас потоковый вывод, что она уже успела сделать какие у тебя входные данные какая история изменений 
По каждому срезу должны быть возможность посмотреть, какая идет работа. 
Срезы должны быть следующие очереди, тикеты, модели, команды, тулы, циклы, дерево циклов, состояние всей системы, бюджеты. 

Принцип: наблюдатель должен понимать, что происходит внутри системы, не вмешиваясь. Если что-то непрозрачно — это дефект системы.

Live view технически: Html страничка, которая общается по вебсокет с непосредственным экзекьютором. 

## Граф тикетов

Тикеты связаны в граф, не только в дерево. Граф отражает реальную структуру зависимостей и порождений.

Типы связей:

```
spawned_by    — кто и когда создал этот тикет
blocks        — этот тикет блокирует другой
blocked_by    — этот тикет заблокирован другим (из dependencies)
related_to    — мягкая информационная связь
evidence_for  — этот тикет предоставляет доказательство для другого
escalates     — этот тикет эскалирует проблему из другого
```

Граф визуализирован в отчёте. Видно, как дерево росло: кто создал что и почему.

## Сценарий в Phase 3

Сценарий верхнего уровня больше не описывает весь граф задач. Он описывает:
- **Задание от человека в plain text** - из них должны получиться:
  - Цели
  - Необходимые условия для достижения цели
- **Бюджет** — Размер бюджета, заданный человеком
- **Доступы** - куда модель имеет доступ (например репозиторий)
- **Верификация** — Что нужно проверить командами или моделью перед тем, как сказать что задача решена


Остальная структура тикетного дерева строится динамически в процессе.

## Аудит

Phase 3 сохраняет принцип Phase 2: всё прозрачно и воспроизводимо.

Каждое событие — явная запись:

- Создание тикета → trace event (`ticket.created`) с actor, reason, evidence
- Переход статуса тикета → trace event (`ticket.status_changed`) с actor и доказательством
- Назначение роли на тикет → trace event (`ticket.assigned`)
- Завершение фазы базового цикла → trace event по каждой фазе
- Создание роли мета-ролью → trace event (`role.created`)

Весь граф тикетов сериализуется в JSON и доступен для инспекции. Replay работает на уровне тикета (не всего сценария).

## Базовые инструменты роли (toolset)

Кроме domain-специфичных инструментов, у каждой роли есть базовые:

```
read_context     — читать разрешённые части контекста
write_context    — писать в разрешённые части контекста
create_ticket    — создать новый тикет (в рамках can_create_ticket_types)
update_ticket    — обновить evidence / findings текущего тикета
request_context  — запросить дополнительный контекст у parent cycle
request_decision — запросить решение у parent или человека
```

Инструменты создания роли доступны только мета-роли:

```
create_role      — создать новую роль с параметрами
```

                               ▼

## Чего Phase 3 не делает

- Не строит distributed runtime (всё выполняется локально)
- Не строит production web app
- Не делает visual GUI конструктор
- Не делает полный sandboxing (filesystem allowlists достаточно)
- Не оптимизирует сложный parallel DAG scheduling
- Не делает semantic diff между запусками

## Приоритеты реализации

1. **Базовый цикл как явный контракт** — четыре фазы стандартизованы в runtime
2. **Тикет = запуск цикла** — полный lifecycle ticket → cycle → result → evidence
3. **create_ticket tool** — модель может создавать тикеты явным вызовом инструмента
4. **Ролевая система** — роли с явными правами, инструментами, очередями
5. **Очереди** — тикеты маршрутизируются в очереди, роли забирают из очередей и работают внутри очереди
6. **Live visibility** — real-time view на дерево тикетов и активные циклы
7. **Граф тикетов** — полноценный граф со всеми типами связей в отчёте
8. **Создатель ролей** — роль, создающая роли
9. **Базовые роли** — описание существующих ролей
10. **Перебор моделей и поиск наилучшей** — ведём score результативности моделей
