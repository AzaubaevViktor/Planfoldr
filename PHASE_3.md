# Phase 3: Динамическая оркестрация с ролями и живым деревом тикетов

## Суть

Phase 2 доказала главный принцип: модель — не центр управления. Детерминированный runtime управляет флоу, модель — исполнитель внутри задач.

Phase 3 делает следующий шаг: runtime становится субстратом для динамической оркестрации, где исполнение порождает новые исполнения. Тикет — это не запись в базе. Тикет — это цикл. Цикл создаёт тикеты. Тикеты становятся циклами. Всё это наблюдаемо в реальном времени.

## Что не так с Phase 2

1. **Статический граф задач** — весь флоу описан заранее в YAML. В реальной работе задачи появляются по ходу.
2. **Нет ролей** — все модели одинаковы. Нет специализации.
3. **Нет очередей** — модели не забирают работу самостоятельно, они запускаются поштучно.
4. **Вложенные циклы не порождаются динамически** — только статически описанные `nested_cycles`.
5. **Visibility** — отчёт строится после завершения. Нет live view на происходящее.

## Базовый цикл

Базовый цикл — атомарная единица работы. Его грамматика:

```text
1. Context Exploration   — модель читает контекст, понимает текущее состояние
2. Changes               — модель использует инструменты, вносит изменения
3. Command Verification  — запуск тестов / команд, сбор механических доказательств
4. Model Verification    — модель оценивает evidence против критериев тикета
```

Не каждый тикет требует всех четырёх фаз. Тикет типа `research` может состоять только из 1 и 4. Тикет типа `verify` — только из 3 и 4. Но последовательность не нарушается: нельзя верифицировать то, что ещё не изменено.

**Базовый цикл — это выполнение тикета.**

## Тикет как единица исполнения

В Phase 2 тикет — это структура данных. В Phase 3 тикет — это описание одного запуска базового цикла.

Когда тикет переходит в `running`, runtime создаёт для него цикл. Когда цикл завершается — тикет получает статус (`done` / `failed`) и evidence.

Поля тикета:

```yaml
id: auth-security-review-01
title: "Проверить авторизацию на уязвимости"
goal: "Найти и задокументировать дыры в модуле auth"
scope:
  includes: ["src/auth/", "tests/auth/"]
  excludes: ["src/billing/"]
type: security_review
required_role: security
constraints:
  - Не изменять код в этом тикете, только документировать
budget:
  max_tokens: 50_000
  max_iterations: 10
dependencies: [auth-implementation-01]
tools: [read_file, run_tests, grep]
context_access:
  read: [scenario.codebase_summary, ticket.auth-implementation-01.artifacts]
  write: [ticket.self.evidence, ticket.self.findings]
links:
  spawned_by: auth-implementation-01
  may_create: [auth-fix-*, auth-pentest-*]
status: ready
evidence: []
```

## Динамическое создание тикетов

Исполняя тикет, модель может создавать новые тикеты. Это — главное отличие от Phase 2.

Создание тикета — явное действие через `create_ticket` tool. Не скрытое в тексте ответа, не угадываемое из контекста — явный вызов инструмента.

Примеры:

- **Разработчик** делает авторизацию → создаёт тикет `security-review` для роли `security`
- **Планировщик** получает задачу → разбивает на N тикетов для разных ролей
- **Исполнитель** обнаруживает нехватку контекста → создаёт тикет `research` для роли `researcher`
- **Верификатор** находит дыру → создаёт тикет `fix` для роли `developer`
- **Security** находит критическую уязвимость → создаёт `block` на тикет релиза

Каждое создание тикета — trace event с:
- кто создал (роль, цикл, execution id)
- почему (reason, evidence из текущего исполнения)
- связь с создателем (`spawned_by`)

Новые тикеты сразу попадают в очередь своей роли.

## Вложенные циклы

Циклы многоуровневые. Один цикл создаёт пачку дочерних.

```text
Orchestration Cycle  (тикет: plan_auth)
  ├── Execution Cycle  (тикет: auth-implementation-01)
  │     └── Execution Cycle  (тикет: auth-tests-01)    ← создан во время impl
  └── Execution Cycle  (тикет: auth-security-review-01)  ← создан после impl
        └── Execution Cycle  (тикет: auth-fix-xss-01)  ← создан security
```

Правила вложенности:

- `constraints` идут вниз от родителя к потомку
- `facts` (результаты, evidence) идут вверх через явно описанный output
- Parent cycle владеет целостностью дерева; child не может объявить весь проект завершённым
- Бюджет делегируется явно при порождении child цикла
- Parent решает: ждать конкретный child result, ждать все результаты, или продолжать

## Ролевая система

Роль — это не настройки модели. Роль — это исполнитель со специализацией.

```
Role = Model + Persona + Tool Set + Queue Access + Constraints + Budget Scope
```

Поля роли:

```yaml
id: security
title: "Security Reviewer"
persona: |
  Ты специалист по безопасности. Твоя работа — найти уязвимости,
  задокументировать их и создать тикеты на исправление.
model:
  provider: ollama
  name: qwen2.5-coder:14b
  fallback: claude-sonnet-4-6
tools: [read_file, run_tests, grep, create_ticket]
can_work_ticket_types: [security_review, penetration_test, audit]
can_create_ticket_types: [fix, block, escalation]
can_create_roles: false
budget_scope:
  max_tokens_per_ticket: 100_000
```

Ключевые поля:

- `can_work_ticket_types` — какие тикеты эта роль может брать из очереди
- `can_create_ticket_types` — какие тикеты эта роль может создавать
- `can_create_roles` — только у мета-роли

## Очереди

Очередь — typed буфер готовых тикетов для конкретной роли.

```
Queue = Ticket Type + Required Role + Priority Policy
```

Когда тикет переходит в `ready` (зависимости выполнены), он появляется в очереди своей роли. Модель с нужной ролью забирает тикет и начинает цикл.

Одна роль может обслуживать несколько очередей. Одну очередь могут обслуживать несколько экземпляров роли параллельно (для параллельной обработки независимых тикетов).

Примеры очередей для сценария "разработка модуля":

```
developer queue     → code, tests, refactor
security queue      → security_review, audit
research queue      → research, documentation
verification queue  → verify, benchmark
orchestration queue → plan, decompose, review_tree
```

## Мета-роль

Есть роль, которая создаёт другие роли.

Мета-роль (`orchestrator` или выделенная `meta`) умеет:
- Анализировать описание задачи и текущее состояние дерева
- Определять, каких ролей не хватает
- Создавать новые роли с нужными параметрами (`create_role` tool)
- Открывать новые очереди
- Переопределять приоритеты

Мета-роль не берёт work tickets — только orchestration. Её цель: сделать так, чтобы нужные роли существовали и нужные тикеты выполнялись.

Пример: задача нестандартная и требует роли `performance_analyst`. Мета-роль создаёт её на ходу с нужным persona, toolset и budget scope.

## Visibility как первый класс

Visibility — это не опция, это основное свойство системы.

Что должно быть наблюдаемо в реальном времени:

**Дерево тикетов** — live view: статусы, зависимости, текущие исполнители.

**Активные циклы** — что сейчас выполняется, в какой фазе базового цикла:
- `exploring_context` — модель читает, анализирует
- `making_changes` — инструменты работают
- `running_verification` — запущены тесты/команды
- `verifying_results` — модель оценивает evidence

**Очереди** — глубина по каждой очереди, сколько работы ждёт.

**Бюджет** — расход токенов и итераций: по тикету, по роли, по циклу, суммарно.

**Поток событий** — аудит-лог в реальном времени: создание тикетов, переходы статусов, решения, блокировки.

**Evidence** — результаты верификации, артефакты, прямые ссылки из тикетов.

Принцип: наблюдатель должен понимать, что происходит внутри системы, не вмешиваясь. Если что-то непрозрачно — это дефект системы.

Live view технически: polling `report_data.json` из HTML (без backend), либо server-sent events от простого local process. Без сложного стека.

## Граф тикетов

Тикеты связаны в граф, не только в дерево. Граф отражает реальную структуру зависимостей и порождений.

Типы связей:

```
spawned_by    — кто и когда создал этот тикет
blocks        — этот тикет блокирует другой
blocked_by    — этот тикет заблокирован другим (из dependencies)
related_to    — мягкая информационная связь
evidence_for  — этот тикет предоставляет доказательство для другого
escalates     — этот тикет эскалирует проблему из другого
```

Граф визуализирован в отчёте. Видно, как дерево росло: кто создал что и почему.

## Сценарий в Phase 3

Сценарий верхнего уровня больше не описывает весь граф задач. Он описывает:
- **Seed tickets** — начальный набор тикетов для запуска
- **Roles** — доступные роли и их конфигурации
- **Queues** — очереди и политики приоритетов
- **Global budget** — общий бюджет сценария
- **Completion criteria** — когда дерево считается завершённым

```yaml
id: build_auth_module
goal: Реализовать и верифицировать модуль авторизации
budget:
  max_tokens: 1_000_000
  max_iterations: 200
roles:
  - ref: roles/orchestrator.yaml
  - ref: roles/developer.yaml
  - ref: roles/security.yaml
  - ref: roles/verifier.yaml
seed_tickets:
  - id: plan-auth-decomposition
    title: Декомпозировать задачу авторизации
    type: orchestration
    required_role: orchestrator
    goal: Создать тикеты для реализации, тестирования и security review
completion:
  when: ticket_tree_fully_verified
  verifier_role: security
  require_evidence: true
```

Остальная структура тикетного дерева строится динамически в процессе.

## Детерминизм и trace в Phase 3

Phase 3 сохраняет принцип Phase 2: всё прозрачно и воспроизводимо.

Каждое событие — явная запись:

- Создание тикета → trace event (`ticket.created`) с actor, reason, evidence
- Переход статуса тикета → trace event (`ticket.status_changed`) с actor и доказательством
- Назначение роли на тикет → trace event (`ticket.assigned`)
- Завершение фазы базового цикла → trace event по каждой фазе
- Создание роли мета-ролью → trace event (`role.created`)

Весь граф тикетов сериализуется в JSON и доступен для инспекции. Replay работает на уровне тикета (не всего сценария).

## Базовые инструменты роли (toolset)

Кроме domain-специфичных инструментов, у каждой роли есть базовые:

```
read_context     — читать разрешённые части контекста
write_context    — писать в разрешённые части контекста
create_ticket    — создать новый тикет (в рамках can_create_ticket_types)
update_ticket    — обновить evidence / findings текущего тикета
request_context  — запросить дополнительный контекст у parent cycle
request_decision — запросить решение у parent или человека
```

Инструменты создания роли доступны только мета-роли:

```
create_role      — создать новую роль с параметрами
```

## Переходы статусов тикета

```
           [dependency resolved]         [role picks up]
 blocked ──────────────────────► ready ──────────────────► running
                                                               │
                              ┌─────────────────┬─────────────┤
                              ▼                 ▼             ▼
                         needs_review         done          failed
                              │
                    [reviewer approves]
                              │
                              ▼
                             done
```

`cancelled` — из любого состояния по явному решению parent cycle или мета-роли.

## Чего Phase 3 не делает

- Не строит distributed runtime (всё выполняется локально)
- Не строит production web app
- Не делает автоматический выбор модели (explicit model config в роли)
- Не делает полный scenario replay (только ticket-level replay)
- Не делает visual GUI конструктор
- Не делает полный sandboxing (filesystem allowlists достаточно)
- Не оптимизирует сложный parallel DAG scheduling
- Не делает semantic diff между запусками

## Приоритеты реализации

1. **Базовый цикл как явный контракт** — четыре фазы стандартизованы в runtime
2. **Тикет = запуск цикла** — полный lifecycle ticket → cycle → result → evidence
3. **create_ticket tool** — модель может создавать тикеты явным вызовом инструмента
4. **Ролевая система** — роли с явными правами, инструментами, очередями
5. **Очереди** — тикеты маршрутизируются в очереди, роли забирают из очередей
6. **Live visibility** — real-time view на дерево тикетов и активные циклы
7. **Граф тикетов** — полноценный граф со всеми типами связей в отчёте
8. **Мета-роль** — роль, создающая роли
