# Phase 4: Recompile — Сущности и связи Phase 3

## Квест 2: Декомпозиция на сущности

---

### 1. Ticket (Тикет)

**2.1 Что делает**
Описывает единицу исполнения: одну задачу, которую нужно выполнить в рамках одного запуска базового цикла. Хранит цель, критерии выполнения, доступы, бюджет и историю.

**2.2 Какие проблемы решает**
- Недетерминированность модели: тикет фиксирует цель и критерии проверки, не даёт модели «срезать углы»
- Отсутствие воспроизводимости: вся история изменений и evidence хранится в тикете
- Отсутствие управляемости: статус тикета — единственный источник правды о прогрессе задачи

**2.3 Цель**
Быть явным, неизменяемым (в части цели) контрактом между постановщиком задачи и исполнителем. Тикет — это одновременно задание и аудит его выполнения.

**2.4 Ограничения**
- Тикет создаётся только через явный вызов `create_ticket` (типы: security-review, research, fix, block и др.)
- Тикет не может объявить себя выполненным самостоятельно — только через прохождение всех обязательных проверок
- Цель тикета не меняется после создания (только через явный процесс пересмотра)
- Тикет принадлежит одной роли-исполнителю, не нескольким одновременно
- Child-тикет не может закрыть parent-тикет
- Тикет не содержит бизнес-логику — только описание и данные

**2.5 Свойства**
- `id`: `<роль>-<номер>`
- `title`: название
- `status`: incoming → blocked → ready → running → done/failed/needs_review
- `goal`: цель тикета
- `checks`: набор обязательных проверок (Command + Model Verification)
- `evidence`: результаты предыдущих проверок
- `accesses`: доступы (к контексту, файлам, базе знаний)
- `budget`: выделенный бюджет
- `dependencies`: список тикетов-зависимостей
- `tools`: разрешённые инструменты
- `metadata`: кто/когда создал, история изменений
- `comments`: список комментариев с указанием автора, времени, призываемой роли, текста

**2.6 Что должен делать (признак полной реализации)**
- Тикет проходит все статусы по явным правилам
- При переходе в `running` автоматически создаётся цикл
- При провале после N попыток переходит в `failed` и модель получает штраф
- Любое изменение тикета записывается в аудит-лог как trace event
- Можно создать child-тикет через `create_ticket` tool
- Комментарии поддерживают призыв роли (`@role`)
- Тикет сериализуется в JSON полностью

**2.7 Что не должен делать**
- Не содержать логику исполнения
- Не запускать модели напрямую
- Не изменять другие тикеты
- Не принимать решения о приоритете (это роль управляющего)
- Не знать о модели, которая его выполняет

### 2. Cycle (Базовый цикл)

**2.1 Что делает**
Атомарная единица работы над тикетом. Выполняет четыре фазы: Context Exploration → Changes → Command Verification → Model Verification. Порождается из тикета, завершается с его статусом.

**2.2 Какие проблемы решает**
- Модели не хватает структуры: явный контракт фаз не позволяет пропустить верификацию
- Модель «забывает» инструкции при большом контексте: цикл перезадаёт роль и цель на каждой фазе
- Вложенные задачи не порождаются динамически: цикл может создать child-циклы через `create_ticket`
- Непрозрачность работы модели: каждая фаза — отдельный observable шаг

**2.3 Цель**
Детерминированно управлять исполнением тикета: гарантировать, что модель прошла каждую фазу в нужном порядке и Evidence собраны до перехода в `done`.

**2.4 Ограничения**
- Фазы выполняются строго последовательно: нельзя верифицировать то, что ещё не изменено
- Не каждый тикет требует всех четырёх фаз (e.g. `research` — только Context + Model Verification; `verify` — только Command + Model Verification), но порядок не нарушается
- Цикл не может напрямую изменить статус другого тикета
- Цикл не может объявить весь проект завершённым: parent-цикл владеет целостностью дерева выполнения
- Child цикл получает делегированный бюджет — не может превысить его

**2.5 Свойства**
- `execution_id`: уникальный идентификатор запуска
- `ticket_id`: тикет, который выполняется
- `role`: роль, под которой работает цикл
- `model`: модель, назначенная на этот запуск
- `phase`: текущая фаза (context_exploration / changes / command_verification / model_verification)
- `local_memory`: временная память, исчезает после завершения
- `budget_used`: израсходованный бюджет
- `parent_cycle_id`: ссылка на родителя (если child)
- `spawned_tickets`: список созданных child-тикетов

**2.6 Что должен делать (признак полной реализации)**
- Создаётся автоматически при переходе тикета в `running`
- Каждая фаза завершается trace event-ом
- После завершения всех фаз тикет переходит в `done` или `needs_review`
- При исчерпании бюджета возвращает `budget_exceeded`, не убивает процесс жёстко
- Constraints передаются вниз к child-циклам, facts идут вверх через output
- Локальная память не утекает в другие циклы

**2.7 Что не должен делать**
- Не пропускать фазы (даже если модель «считает» это лишним)
- Не писать напрямую в тикеты других циклов
- Не менять constraints родителя
- Не превышать делегированный бюджет без явного запроса
- Не хранить state между разными запусками

### 3. Role (Роль)

**2.1 Что делает**
Специализирует модель: задаёт системный промпт, scope доступных инструментов и область ответственности. Роль — это контракт между системой и LLM о том, как та должна работать в конкретном контексте.

**2.2 Какие проблемы решает**
- Все модели одинаковы и не специализированы (проблема Phase 2)
- Модель может выйти за рамки задачи: роль ограничивает toolset и scope
- Повторяемость поведения: роль задаёт предсказуемые паттерны работы

**2.3 Цель**
Быть контейнером специализации, который можно переиспользовать для множества тикетов и очередей, и расширять через добавление очередного промпта и скоупа.

**2.4 Ограничения**
- Роль не может сама себя изменить
- Роль не может создать другую роль — это исключительное право `birthgiver`
- Роль не знает, какая конкретно модель её выполняет — это решает runtime
- Инструменты роли ограничены: нет доступа к инструментам вне разрешённого scope

**2.5 Свойства**
- `id`: уникальный идентификатор
- `prompt`: системный промпт специализации
- `toolset`: разрешённые инструменты (base + domain)
- `queue_prompts`: дополнительные промпты, подмешиваемые от очереди
- `queue_scopes`: дополнительные tool-scopes от очереди
- `can_create_ticket_types`: типы тикетов, которые роль может создавать
- `score_history`: статистика успешности по ролям и моделям

**2.6 Что должна делать (признак полной реализации)**
- Роль корректно ограничивает доступные инструменты в рамках цикла
- Промпт роли подмешивается к промпту очереди при назначении
- Роль можно переиспользовать в нескольких очередях
- Несколько экземпляров одной роли могут работать параллельно
- Статистика роли обновляется после каждого тикета

**2.7 Что не должна делать**
- Не выполнять тикеты напрямую (это делает цикл + модель)
- Не принимать решения о приоритете тикетов
- Не изменять структуру других ролей
- Не иметь доступа к чужим локальным памятям

### 4. Queue (Очередь)

**2.1 Что делает**
Группирует тикеты по направлению работы и обеспечивает их маршрутизацию к нужным ролям. Содержит тикеты в разных статусах и поддерживает параллельную обработку.

**2.2 Какие проблемы решает**
- Нет специализации по направлениям: очередь изолирует `developer`, `security`, `research` потоки
- Нет механизма параллельной обработки: несколько экземпляров роли могут работать внутри одной очереди
- Нет структурирования входящего потока: управляющий приоритизирует внутри очереди

**2.3 Цель**
Быть буфером и маршрутизатором между тикетами и ролями. Очередь отвечает за то, чтобы правильная роль взяла правильный тикет в правильный момент.

**2.4 Ограничения**
- Очередь не исполняет тикеты сама
- Очередь не меняет цели тикетов
- Одна очередь не может захватить тикеты другой очереди
- Очередь не принимает `declined` тикеты обратно без явного пересмотра

**2.5 Свойства**
- `id`, `name`: идентификатор и название
- `description`: в каких случаях и как обращаться к этой очереди
- `roles`: прикреплённые роли (управляющий + исполнитель)
- `tickets`: список тикетов по статусам
- `template`: шаблон создания тикетов (JSON)
- `extra_prompt`: дополнительный промпт, добавляемый к ролям
- `extra_scope`: дополнительный tool scope для ролей

**2.6 Что должна делать (признак полной реализации)**
- Новый тикет с соответствующей ролью попадает в очередь автоматически при переходе в `ready`
- Управляющий может переводить тикеты из `incoming` в `in_queue, priority` или `declined, cause`
- Параллельное выполнение: несколько исполнителей берут независимые тикеты одновременно
- Шаблон тикета доступен для редактирования управляющим

**2.7 Что не должна делать**
- Не запускать циклы напрямую
- Не оценивать качество выполнения тикетов (это верификатор и аудит)
- Не управлять бюджетами напрямую

---

### 5. Queue Manager (Управляющий очередью)

**2.1 Что делает**
Роль, обслуживающая конкретную очередь: просматривает входящий поток тикетов, расставляет приоритеты, отклоняет нерелевантные задачи и меняет шаблон очереди.

**2.2 Какие проблемы решает**
- Без фильтрации любая модель создаёт тикеты бесконтрольно: управляющий — это gatekeeping
- Нет приоритизации: управляющий явно расставляет priority
- Шаблон очереди может устареть: управляющий его адаптирует

**2.3 Цель**
Обеспечить, чтобы исполнители всегда работали с осмысленными задачами в правильном порядке. Отклонить всё, что не соответствует цели очереди и проекта.

**2.4 Ограничения**
- Управляющий не исполняет тикеты сам
- Управляющий не изменяет цель тикетов, только их приоритет и статус `incoming → in_queue / declined`
- Declined тикеты видны только управляющему (не исполнителю)
- Управляющий не может создавать роли (только `birthgiver`)

**2.5 Свойства**
Унаследует от Role плюс:
- `queue_id`: привязка к очереди
- `triage_prompt`: специализированный промпт для приоритизации

**2.6 Что должен делать (признак полной реализации)**
- Периодически просматривает `incoming` тикеты в своей очереди
- Переводит тикет в `in_queue, priority: N` или `declined, cause: <reason>`
- Может редактировать шаблон очереди
- Может призывать `@birthgiver` если очереди не хватает ролей
- Каждое решение записывается в аудит

**2.7 Что не должен делать**
- Не запускать циклы над тикетами
- Не блокировать тикеты из других очередей
- Не менять goals и checks тикета
- Не начислять баллы моделям

### 6. Executor (Копатель)

**2.1 Что делает**
Роль-исполнитель: забирает тикет из очереди и запускает над ним базовый цикл. Непосредственно делает работу — пишет код, исследует, верифицирует.

**2.2 Какие проблемы решает**
- Модели не специализированы: копатель ограничен ролевым промптом и toolset-ом
- Нет механизма «взятия в работу»: копатель явно переводит тикет в `running`

**2.3 Цель**
Выполнить конкретный тикет: достичь его цели, пройти все проверки и вернуть evidence.

**2.4 Ограничения**
- Один копатель — один тикет за раз
- Копатель не управляет очередью
- Копатель не может одобрить свою собственную работу в Model Verification
- Не может создавать роли

**2.5 Свойства**
Унаследует от Role плюс:
- `current_ticket_id`: тикет в работе
- `attempt_count`: количество попыток текущего тикета

**2.6 Что должен делать (признак полной реализации)**
- Забирает `in_queue` тикет с максимальным приоритетом
- Запускает базовый цикл
- Создаёт child-тикеты через `create_ticket` при обнаружении новых задач
- После N неудачных попыток тикет переходит в `failed`
- Результаты и evidence записываются в тикет

**2.7 Что не должен делать**
- Не изменять приоритеты очереди
- Не принимать решения об отклонении тикетов
- Не работать с тикетами других очередей без явного перенаправления

### 7. Birthgiver (Создатель ролей)

**2.1 Что делает**
Мета-роль: анализирует нехватку ролей в системе, создаёт новые роли, открывает очереди, привязывает роли к очередям.

**2.2 Какие проблемы решает**
- Статическая конфигурация ролей: динамические задачи требуют динамических ролей
- Дублирование ролей: birthgiver проверяет существующие роли перед созданием

**2.3 Цель**
Быть единственной точкой расширения ролевой системы. Гарантировать, что каждая новая роль обоснована, не дублирует существующую и правильно настроена.

**2.4 Ограничения**
- Создание роли требует явного тикета к birthgiver — не автоматически
- Birthgiver может отказать в создании роли с обоснованием
- Созданные роли не могут рекурсивно создавать birthgiver
- Birthgiver не исполняет тикеты других типов

**2.5 Свойства**
- `create_role` tool: исключительный инструмент
- `role_registry`: доступ к реестру всех существующих ролей
- `queue_registry`: доступ к реестру всех очередей

**2.6 Что должен делать (признак полной реализации)**
- Проверяет реестр: если роль уже есть — связывает с существующей
- Если роль не нужна — отказывает с причиной
- Если роль нужна — создаёт очередь + роли (управляющий + исполнитель) + toolset + budget scope
- Каждое создание роли — trace event `role.created`
- Призыв к несуществующей роли через `@role` попадает к birthgiver как incoming тикет

**2.7 Что не должен делать**
- Не исполнять обычные задачи разработки
- Не управлять приоритетами очередей
- Не начислять баллы
- Не менять цели существующих ролей

### 8. Model (LLM / Модель)

**2.1 Что делает**
Языковая модель — непосредственный исполнитель работы внутри цикла. Генерирует текст, вызывает инструменты, принимает решения в рамках роли и тикета.

**2.2 Какие проблемы решает**
- Не все модели одинаково справляются с задачами: система ведёт статистику и выбирает лучшую
- Дорогие модели не нужны везде: подбирается минимально достаточная

**2.3 Цель**
Быть заменяемым исполнителем. Модель — не центр управления, а инструмент runtime-а. Runtime выбирает модель, runtime контролирует флоу.

**2.4 Ограничения**
- Модель не управляет своим выбором — это делает runtime по score
- Модель не знает о других параллельных циклах
- Модель не видит чужую локальную память
- Модель не может изменить свою роль во время выполнения
- Модель не может превысить `max_tokens_per_ticket`

**2.5 Свойства**
- `id`: идентификатор (ollama/api/subscription)
- `provider`: ollama / openai / anthropic / subscription
- `parameter_count`: базовый счёт (больше параметров → выше базовый ранг)
- `score`: агрегированный балл (общий + по роли + по типу задачи)
- `cost_per_token`: для API-моделей
- `resource_usage`: cpu/ram/gpu*ram*time для локальных

**2.6 Что должна делать (признак полной реализации)**
- Модель выбирается runtime-ом на основе score для конкретной роли и типа задачи
- После 2-3 провалов на одном типе задачи — автоматически берётся другая/более мощная
- Использование токенов и денег логируется на уровне каждого запроса
- Для ollama: GPU*RAM*time считается через `ollama ps`
- Score обновляется после каждого завершённого тикета

**2.7 Что не должна делать**
- Не управлять флоу — это runtime
- Не выбирать себя для следующего тикета
- Не знать о скоринговой системе (это внутренность runtime)
- Не иметь доступа к тикетам за пределами текущего цикла (без явного разрешения)

### 9. Budget (Бюджет)

**2.1 Что делает**
Отслеживает и ограничивает потребление ресурсов на уровне проекта, очереди, тикета и цикла. Обновляется в реальном времени по ходу работы.

**2.2 Какие проблемы решает**
- Неконтролируемые расходы на модели и команды
- Невозможность сравнить эффективность разных моделей без учёта стоимости

**2.3 Цель**
Гарантировать, что система не превысит ресурсный лимит, и обеспечить данные для оптимизации стоимости.

**2.4 Ограничения**
- Бюджет делегируется явно при порождении child-цикла
- При превышении бюджета — система даёт закончить ongoing-таски, но не запускает новые
- Бюджет не может быть тихо проигнорирован: превышение → trace event + `budget_exceeded`

**2.5 Свойства**
Для команд: `cpu_usage`, `ram_usage`, `command_runs`
Для файлов: `file_changes`, `lines_added`, `lines_removed`
Для моделей: `tokens_used`, `money_spent`, `api_requests`, `gpu_ram_hours`
Общие: `queues_created`, `roles_created`, `tickets_created`

**2.6 Что должен делать (признак полной реализации)**
- Бюджет обновляется после каждой операции (не в конце тикета)
- Превышение бюджета → soft stop: ongoing завершаются, новые не стартуют → `budget_exceeded`
- Статистика по модели хранится отдельно для двух параллельных тикетов с одной моделью
- Бюджет сериализуется в аудит

**2.7 Что не должен делать**
- Не убивать процессы жёстко (только soft stop)
- Не блокировать аудит-логирование при превышении бюджета
- Не считать бюджет только в конце — только real-time

---

### 10. Visibility (Наблюдаемость)

**2.1 Что делает**
Делает всю работу системы прозрачной в реальном времени через два режима: потоковый лог (что сейчас происходит) и общее состояние (граф очередей, тикетов, моделей).

**2.2 Какие проблемы решает**
- Чёрный ящик: наблюдатель не понимает, что происходит внутри
- Отчёт только после завершения (проблема Phase 2)
- Невозможность отладить поведение модели без остановки системы

**2.3 Цель**
Любой наблюдатель в любой момент должен понять: какая модель, с какими данными, в рамках какого тикета/цикла/очереди делает что — без вмешательства в процесс.

**2.4 Ограничения**
- Visibility только читает — не может изменять состояние
- Не замедляет исполнение: события пишутся асинхронно
- Не раскрывает секреты/credentials из toolset-а

**2.5 Свойства**
- `streaming_log`: HTML-страница с потоковыми событиями (thinking → output → tool calls → results)
- `state_view`: вторая страница с live-срезами: очереди / тикеты / модели / команды / тулы / циклы / дерево циклов / бюджеты
- `websocket`: соединение между HTML и executor-ом
- Срезы: drill-down от очереди → тикет → цикл → модель → текущий поток вывода

**2.6 Что должна делать (признак полной реализации)**
- Каждое событие (фаза цикла, вызов тула, создание тикета) отображается в потоковом логе
- Для каждого execution: входные данные, контекст, доступные тулы — в раскрываемом блоке (details/summary)
- Для tool call: краткий текст + детали внутри
- State view обновляется в реальном времени по WS
- Непрозрачное событие = дефект системы

**2.7 Что не должна делать**
- Не влиять на исполнение (read-only)
- Не буферизировать события до завершения тикета
- Не скрывать промежуточные состояния

---

### 11. Ticket Graph (Граф тикетов)

**2.1 Что делает**
Хранит и визуализирует связи между тикетами: кто кого породил, кто кого блокирует, кто предоставляет evidence для кого.

**2.2 Какие проблемы решает**
- Неясная история принятия решений: граф показывает, почему появился тот или иной тикет
- Скрытые зависимости: граф делает их явными и управляемыми

**2.3 Цель**
Быть единственным источником правды о структуре зависимостей и истории порождения тикетов.

**2.4 Ограничения**
- Граф не циклический в части `blocked_by` / `blocks` (нет deadlock-ов по зависимостям)
- Связи только явные — не угадываются из текста
- Граф не изменяется задним числом без trace event-а

**2.5 Свойства**
Типы связей:
- `spawned_by`: кто и когда создал тикет
- `blocks`: этот тикет блокирует другой
- `blocked_by`: этот тикет заблокирован другим
- `related_to`: мягкая информационная связь
- `evidence_for`: этот тикет предоставляет доказательство для другого
- `escalates`: эскалация проблемы из другого тикета

**2.6 Что должен делать (признак полной реализации)**
- Граф строится автоматически при создании тикета (ссылка на `spawned_by`)
- Граф визуализирован в отчёте
- Сериализуется в JSON
- Replay возможен на уровне любого тикета по данным графа

**2.7 Что не должен делать**
- Не управлять исполнением (это runtime и очереди)
- Не создавать неявные связи из текста тикетов
- Не удалять связи — только добавлять (история неизменна)

---

### 12. Audit (Аудит)

**2.1 Что делает**
Записывает все значимые события системы как неизменяемые trace events. Гарантирует воспроизводимость и инспектируемость любого решения.

**2.2 Какие проблемы решает**
- Непрозрачность принятых решений: любое изменение статуса, создание роли, провал тикета — в аудите
- Невозможность разобраться в инцидентах: replay по аудиту позволяет воспроизвести любой момент

**2.3 Цель**
Быть полной, append-only историей всего, что произошло в системе. Аудит — не опция, это обязательная составляющая каждого события.

**2.4 Ограничения**
- Аудит только пишется, не редактируется
- Не останавливается при `budget_exceeded`
- Не фильтрует события по умолчанию

**2.5 Свойства**
Типы событий:
- `ticket.created`: actor, reason, evidence, spawned_by
- `ticket.status_changed`: actor, from, to, proof
- `ticket.assigned`: role, model, execution_id
- `cycle.phase_completed`: фаза, duration, tokens
- `role.created`: создатель, причина, параметры
- `budget.exceeded`: тип ресурса, лимит, факт
- `model.score_updated`: модель, роль, delta, причина

**2.6 Что должен делать (признак полной реализации)**
- Каждое событие атомарно записывается до выполнения действия (или после, с rollback-записью)
- Полный аудит сериализуется в JSON
- Возможен replay на уровне тикета
- Аудит доступен через Visibility

**2.7 Что не должен делать**
- Не участвовать в бизнес-логике (только запись)
- Не блокировать основной поток при записи
- Не удалять или изменять записи

---

### 13. Scenario (Сценарий)

**2.1 Что делает**
Описывает задание от человека: plain-text цель, бюджет, доступы и критерии верификации. Является входной точкой для запуска всей системы.

**2.2 Какие проблемы решает**
- Статический граф задач в Phase 2: сценарий теперь только описывает начальную точку, не весь граф
- Человек должен явно задать бюджет и доступы, чтобы система не вышла за рамки

**2.3 Цель**
Быть мостом между намерением человека и динамической системой тикетов. Из сценария рождается верхний цикл, который порождает всё остальное.

**2.4 Ограничения**
- Сценарий не описывает весь граф задач — только цель и ограничения
- Сценарий задаётся человеком, не модифицируется моделями
- Сценарий не может быть запущен без явного бюджета и доступов

**2.5 Свойства**
- `goal_text`: plain text задание от человека
- `goals`: сформулированные цели (из goal_text через Context Exploration верхнего цикла)
- `constraints`: ограничения
- `budget`: размер бюджета
- `accesses`: доступы (репозиторий, файловая система и т.д.)
- `verification`: что проверить командами/моделью перед завершением

**2.6 Что должен делать (признак полной реализации)**
- Запуск сценария создаёт верхний цикл
- Из plain text формулируются цели и ограничения через Context Exploration
- `@human` — выделенная роль для запроса дополнительной информации у человека
- Верификация на уровне сценария — это финальный gate

**2.7 Что не должен делать**
- Не описывать полный граф тикетов заранее
- Не управлять ролями и очередями напрямую
- Не изменяться после запуска

---

### 14. Knowledge Base (База знаний)

**2.1 Что делает**
Хранит разделяемый контекст, доступный нескольким циклам в рамках проекта. В отличие от локальной памяти цикла — персистентна.

**2.2 Какие проблемы решает**
- Циклы изолированы по локальной памяти: нужен механизм для разделения знаний
- Контекст теряется между тикетами без явного хранения

**2.3 Цель**
Быть управляемым общим хранилищем знаний, где каждый цикл читает/пишет только то, что ему разрешено.

**2.4 Ограничения**
- Доступ к базе знаний ограничен scope-ом роли/тикета
- Запись в базу знаний — явное действие через `write_context` tool
- Запись логируется в аудит

**2.5 Свойства**
- Разделяется на секции с access control по ролям
- Версионируется (история изменений)

**2.6 Что должна делать (признак полной реализации)**
- Роль/цикл читает только разрешённые секции
- Запись в разрешённую секцию логируется
- Изменения в базе знаний видны в Visibility

**2.7 Что не должна делать**
- Не хранить локальную память циклов
- Не давать доступ вне разрешённого scope
- Не быть единственным местом хранения тикетов и аудита

---

### 15. Toolset

**2.1 Что делает**
Определяет набор инструментов, доступных роли в рамках цикла. Разделяется на базовые (у всех ролей) и domain-специфичные.

**2.2 Какие проблемы решает**
- Модель не должна иметь произвольный доступ к системе: toolset ограничивает blast radius
- Нужна согласованность: `согласователь тулов` одобряет доступ к инструментам

**2.3 Цель**
Минимально необходимый набор инструментов для роли — принцип least privilege.

**2.4 Ограничения**
- Изменение toolset-а требует согласования с особой ролью (`согласователь тулов`)
- Базовые инструменты одинаковы для всех ролей
- `create_role` tool — исключительно у birthgiver

**2.5 Свойства**
Базовые инструменты (у всех ролей):
- `read_context`, `write_context`
- `create_ticket`, `update_ticket`
- `request_context`, `request_decision`

Domain инструменты: специфичны для роли (bash, code editor, search и т.д.)

**2.6 Что должен делать (признак полной реализации)**
- Роль не может вызвать инструмент вне своего toolset-а
- Вызов неразрешённого инструмента → trace event + отказ
- Toolset документирован и версионирован

**2.7 Что не должен делать**
- Не давать доступ к `create_role` никому кроме birthgiver
- Не изменяться во время активного цикла

---

### 16. Score System (Скоринг моделей)

**2.1 Что делает**
Ведёт статистику результативности языковых моделей по осям: общий балл, балл по роли, балл по типу задачи. Используется для выбора модели при следующем запуске.

**2.2 Какие проблемы решает**
- Нет обратной связи о качестве моделей: система слепо использует одну модель
- Поставщики могут ухудшить модель после дообучения: score выявит деградацию

**2.3 Цель**
Автоматически улучшать выбор модели на основе накопленного опыта, без ручной настройки.

**2.4 Ограничения**
- Скоринг не принимает решения — только предоставляет данные runtime-у
- Балл не может быть изменён напрямую моделью (только runtime после верификации)
- Начальный балл = f(parameter_count), но может быть переопределён данными

**2.5 Свойства**
- `global_score`: общий балл по всем задачам
- `role_score`: балл по конкретной роли
- `task_type_score`: балл по типу задачи
- Критерии начисления: выполнен тикет (+), верифицирован (+), экономия бюджета (+), быстрое выполнение (+)
- Критерии вычитания: failed тикет (со штрафом за сложность) (-), бюджет исчерпан (-), >80% токенов (-), ложная верификация (-), излишние child-тикеты (-)

**2.6 Что должен делать (признак полной реализации)**
- После каждого тикета score обновляется автоматически
- После 2-3 провалов одного типа → runtime берёт другую/более мощную модель
- Score хранится в аудите
- Статистика доступна через Visibility

**2.7 Что не должен делать**
- Не выбирать модель напрямую (это runtime по данным score)
- Не давать модели доступ к своей собственной оценке
- Не сбрасывать историю при смене провайдера

## Квест 3: Связи между парами сущностей

---

### Ticket ↔ Cycle

**3.1 Как взаимодействуют**
- Ticket → Cycle: переход в статус `running` автоматически создаёт Cycle для этого тикета; тикет передаёт циклу цель, checks, toolset, budget, accesses
- Cycle → Ticket: по завершении цикл обновляет статус тикета (`done`/`failed`/`needs_review`), записывает evidence и spawned_tickets

**3.2 Как не должны взаимодействовать**
- Cycle не создаёт тикеты напрямую в обход `create_ticket` tool — только явный вызов инструмента
- Ticket не управляет фазами цикла — фазы контролирует runtime

---

### Ticket ↔ Queue

**3.1 Как взаимодействуют**
- Ticket → Queue: при создании тикет попадает в `incoming` нужной очереди по типу роли
- Queue → Ticket: управляющий переводит тикет из `incoming` в `in_queue, priority` или `declined`
- Когда зависимости тикета выполнены → тикет становится `ready` и доступен для взятия исполнителем

**3.2 Как не должны взаимодействовать**
- Queue не изменяет цель или checks тикета
- Ticket не выбирает очередь самостоятельно — только через роль-назначение

---

### Ticket ↔ Ticket Graph

**3.1 Как взаимодействуют**
- Каждый новый тикет добавляет узел в граф с ссылками `spawned_by`, `blocked_by`, `blocks`
- Граф используется для разрешения зависимостей: тикет переходит в `ready` когда все `blocked_by` = `done`
- При создании связи `evidence_for` или `escalates` — через явное поле тикета

**3.2 Как не должны взаимодействовать**
- Граф не управляет исполнением тикетов — он только отражает состояние
- Тикеты не создают связи `blocks` рекурсивно (deadlock prevention)

---

### Ticket ↔ Audit

**3.1 Как взаимодействуют**
- Каждое изменение тикета (создание, смена статуса, назначение, completion) → trace event в аудите
- Аудит хранит полную историю тикета, включая все попытки и evidence

**3.2 Как не должны взаимодействовать**
- Аудит не изменяет тикет
- Тикет не читает аудит-лог напрямую (только через явный инструмент, если разрешён ролью)

---

### Cycle ↔ Role

**3.1 Как взаимодействуют**
- Role → Cycle: определяет промпт и toolset, с которым работает цикл
- Cycle → Role: обновляет score роли через Score System после завершения
- Несколько циклов могут выполняться параллельно под одной ролью (разные экземпляры)

**3.2 Как не должны взаимодействовать**
- Cycle не изменяет определение роли во время исполнения
- Role не знает о конкретном исполняющем её цикле (stateless определение)

---

### Cycle ↔ Model

**3.1 Как взаимодействуют**
- Runtime назначает Model на Cycle по score для данной роли и типа задачи
- Model генерирует output внутри цикла: thinking, текст, вызовы инструментов
- Cycle передаёт в Model: промпт роли, контекст тикета, toolset
- После завершения: статистика использования модели (токены, деньги) → Budget + Score System

**3.2 Как не должны взаимодействовать**
- Model не выбирает себя для следующего цикла
- Cycle не даёт Model доступ к данным других параллельных циклов
- Model не может изменить toolset во время выполнения

---

### Cycle ↔ Budget

**3.1 Как взаимодействуют**
- Cycle получает делегированный Budget при создании
- Каждая операция внутри цикла (токены, команды, изменения файлов) обновляет Budget в реальном времени
- При достижении лимита: цикл получает `budget_exceeded` сигнал → завершает текущую операцию, не запускает следующую

**3.2 Как не должны взаимодействовать**
- Cycle не может увеличить свой бюджет без явного `request_decision` к parent
- Budget не убивает cycle жёстко — только soft stop

---

### Cycle ↔ Visibility

**3.1 Как взаимодействуют**
- Каждая фаза цикла эмитирует события в Visibility: начало фазы, thinking модели, вызовы инструментов, результаты
- Visibility читает стримирующий вывод модели в реальном времени через WebSocket

**3.2 Как не должны взаимодействуют**
- Visibility не влияет на выполнение цикла
- Cycle не ждёт подтверждения от Visibility

---

### Role ↔ Queue

**3.1 Как взаимодействуют**
- Queue закрепляет за собой роли: управляющий + исполнитель
- Queue подмешивает к базовому промпту роли `queue_prompt` и расширяет `queue_scope`
- Одна роль может обслуживать несколько очередей (разные queue_prompts подмешиваются по контексту)

**3.2 Как не должны взаимодействовать**
- Queue не переопределяет базовый промпт роли, только расширяет
- Роль из одной очереди не видит тикеты другой очереди без явного перенаправления

---

### Role ↔ Birthgiver

**3.1 Как взаимодействуют**
- Birthgiver создаёт новые роли через `create_role` tool
- При призыве несуществующей роли через `@role` — запрос попадает к Birthgiver как `incoming` тикет
- Birthgiver проверяет реестр ролей перед созданием

**3.2 Как не должны взаимодействовать**
- Обычные роли не могут вызвать `create_role`
- Birthgiver не управляет приоритетами или исполнением существующих ролей

---

### Queue Manager ↔ Executor

**3.1 Как взаимодействуют**
- Queue Manager переводит тикет в `in_queue, priority` → Executor видит доступный тикет
- Executor забирает тикет с максимальным приоритетом
- Если Executor возвращает тикет в `needs_review` — Queue Manager видит его в специальном статусе

**3.2 Как не должны взаимодействовать**
- Queue Manager не вмешивается в активный цикл исполнителя
- Executor не меняет приоритеты очереди

---

### Model ↔ Score System

**3.1 Как взаимодействуют**
- Score System получает результаты после каждого цикла: passed/failed, токены, время, budget usage
- Runtime читает Score System при выборе модели для нового цикла
- После 2-3 провалов Score System сигнализирует runtime взять другую модель

**3.2 Как не должны взаимодействовать**
- Model не читает свой score напрямую
- Score System не принимает решения о выборе модели — это runtime по данным score

---

### Budget ↔ Scenario

**3.1 Как взаимодействуют**
- Scenario задаёт верхний Budget: размер и типы ресурсов
- Budget делегируется вниз при создании каждого child-цикла
- Scenario-level верификация — финальный gate после завершения бюджета

**3.2 Как не должны взаимодействовать**
- Scenario не может быть изменён после запуска, даже если Budget исчерпан
- Budget не блокирует финальную верификацию Scenario

---

### Visibility ↔ Audit

**3.1 Как взаимодействуют**
- Audit — источник исторических данных для Visibility
- Visibility отображает аудит-события в потоковом логе в реальном времени
- State view Visibility агрегирует данные из аудита для срезов по очередям/тикетам/моделям

**3.2 Как не должны взаимодействовать**
- Visibility не записывает в аудит
- Аудит не форматирует данные для Visibility — только хранит raw events

---

### Ticket Graph ↔ Queue

**3.1 Как взаимодействуют**
- Когда `blocked_by` тикеты переходят в `done` — граф сигнализирует Queue, что зависимый тикет готов к переводу в `ready`
- Queue использует граф для проверки зависимостей перед разрешением исполнения

**3.2 Как не должны взаимодействовать**
- Queue не изменяет связи в графе
- Граф не управляет приоритизацией внутри очереди

---

### Knowledge Base ↔ Cycle

**3.1 Как взаимодействуют**
- Cycle читает разрешённые секции Knowledge Base в фазе Context Exploration
- Cycle записывает в Knowledge Base через `write_context` tool в фазе Changes
- Изменения в Knowledge Base доступны другим циклам немедленно (в рамках их scope)

**3.2 Как не должны взаимодействовать**
- Cycle не читает секции вне своего scope
- Knowledge Base не хранит локальную память цикла

---

### Birthgiver ↔ Queue

**3.1 Как взаимодействуют**
- Birthgiver создаёт новую очередь вместе с ролями при необходимости
- Birthgiver может подмешивать роли к существующим очередям

**3.2 Как не должны взаимодействовать**
- Birthgiver не управляет тикетами внутри очереди
- Queue не запрашивает создание ролей напрямую (только через тикет к Birthgiver)

---

### Scenario ↔ Cycle (верхний)

**3.1 Как взаимодействуют**
- Scenario порождает верхний Cycle при запуске
- Верхний Cycle уточняет цели сценария в фазе Context Exploration (может запросить `@human`)
- Верхний Cycle создаёт первичные тикеты, которые порождают всё дерево

**3.2 Как не должны взаимодействовать**
- Верхний Cycle не переписывает Scenario
- Scenario не управляет дочерними циклами напрямую

---

### Toolset ↔ Audit

**3.1 Как взаимодействуют**
- Каждое изменение Toolset (добавление/удаление инструментов) → trace event в Audit
- Попытка вызвать запрещённый инструмент → trace event в Audit

**3.2 Как не должны взаимодействовать**
- Audit не влияет на разрешения Toolset
- Toolset не знает о структуре аудита

---

### Score System ↔ Budget

**3.1 Как взаимодействуют**
- Экономия бюджета начисляет бонусные баллы в Score System
- Исчерпание бюджета начисляет тяжёлый штраф
- Budget передаёт данные о расходе токенов и денег в Score System после каждого тикета

**3.2 Как не должны взаимодействовать**
- Score System не влияет на лимиты Budget
- Budget не принимает решений о выборе модели

---

## Квест 4: Сортировка сущностей от базовых к комплексным

Порядок от наиболее независимых (нет зависимостей от других сущностей) к наиболее зависимым (используют многие другие).

### Уровень 0 — Фундамент (ни от кого не зависят)

**1. Audit**
Чисто append-only лог. Не зависит ни от одной другой сущности — только принимает события. Все остальные сущности зависят от него.

**2. Toolset**
Статический реестр инструментов. Определяется конфигурацией, не зависит от runtime-состояния других сущностей.

---

### Уровень 1 — Примитивы данных

**3. Knowledge Base**
Зависит только от Toolset (через `read_context`/`write_context`) и Audit (запись изменений). Сама по себе — просто управляемое хранилище.

**4. Budget**
Зависит от Audit (логирует превышения). Не требует ни моделей, ни циклов для своей структуры. Обновляется событиями от других, но структура независима.

**5. Score System**
Зависит от Audit (хранит историю баллов) и Budget (читает расход). Структура сама по себе проста — таблица оценок по моделям/ролям/задачам.

---

### Уровень 2 — Базовые рабочие единицы

**6. Ticket**
Зависит от: Toolset (список разрешённых тулов), Budget (бюджет тикета), Audit (trace events).
Ticket — первая сущность с lifecycle и бизнес-логикой (статусы, проверки, evidence).

**7. Role**
Зависит от: Toolset (scope доступа), Score System (статистика), Audit.
Роль — контейнер специализации, не зависит от конкретных тикетов или очередей.

---

### Уровень 3 — Структуры управления

**8. Ticket Graph**
Зависит от: Ticket (узлы графа), Audit (история связей).
Граф строится поверх тикетов, отражает их зависимости.

**9. Queue**
Зависит от: Ticket (содержит тикеты), Role (прикреплённые роли), Ticket Graph (проверка зависимостей), Audit.
Очередь — первая сущность, которая координирует несколько других.

---

### Уровень 4 — Специализированные роли

**10. Queue Manager**
Зависит от: Role (наследует), Queue (управляет конкретной очередью), Ticket (приоритизирует), Audit.
Экземпляр Role, специализированный для управления очередью.

**11. Executor (Копатель)**
Зависит от: Role (наследует), Queue (берёт тикеты), Ticket (исполняет), Toolset, Budget, Audit.
Экземпляр Role, специализированный для исполнения тикетов.

**12. Birthgiver (Создатель ролей)**
Зависит от: Role (создаёт и управляет реестром), Queue (создаёт очереди), Toolset (назначает scope), Audit.
Мета-роль: зависит от всей ролевой системы, чтобы расширять её.

---

### Уровень 5 — Исполнение

**13. Model (LLM)**
Зависит от: Role (получает промпт и scope), Budget (ограничивает токены), Score System (по ней выбирают), Audit.
Модель — заменяемый исполнитель внутри цикла. Зависит от многих, но сама ни от чего не зависит структурно.

**14. Cycle (Базовый цикл)**
Зависит от: Ticket (создаётся из тикета), Role (промпт и toolset), Model (исполнитель), Budget (делегированный), Knowledge Base (контекст), Toolset, Audit, Ticket Graph (порождает связи).
Cycle — оркестратор исполнения. Самая операционно-насыщенная сущность уровня исполнения.

---

### Уровень 6 — Наблюдаемость

**15. Visibility**
Зависит от: Cycle (стримит события), Ticket (статусы), Queue (срезы), Model (live output), Budget (live бюджет), Audit (история), Score System (статистика).
Visibility ничего не создаёт — только агрегирует и отображает. Зависит от почти всего.

---

### Уровень 7 — Вход и верхний контроль

**16. Scenario**
Зависит от: Cycle (порождает верхний цикл), Budget (задаёт верхний лимит), Toolset (задаёт доступы), Audit (верхний gate).
Scenario — внешний вход. Зависит от всей системы концептуально, но структурно минимален: plain text + параметры.

---

### Итоговая таблица зависимостей

```
Уровень 0: Audit, Toolset
Уровень 1: Knowledge Base, Budget, Score System
Уровень 2: Ticket, Role
Уровень 3: Ticket Graph, Queue
Уровень 4: Queue Manager, Executor, Birthgiver
Уровень 5: Model, Cycle
Уровень 6: Visibility
Уровень 7: Scenario
```

Реализовывать в этом же порядке: начиная с Audit и Toolset, заканчивая Scenario. Каждый следующий уровень строится поверх предыдущих.

---

## Квест 5: Общий флоу работы между всеми сущностями

### Шаг 1 — Запуск: Scenario → верхний Cycle

Человек создаёт **Scenario**: plain-text цель, бюджет, доступы, критерии верификации.

Runtime создаёт **верхний Cycle** с выделенным **Budget** и пустым **Ticket Graph**.

**Birthgiver** автоматически создаёт базовый набор ролей и очередей: `orchestration`, `developer`, `research`, `verification`. Каждая очередь получает двух агентов: **Queue Manager** и **Executor**. Каждый агент — это **Role** с промптом + **Toolset**.

```
Scenario
  └─► верхний Cycle (budget=верхний, role=orchestrator)
        └─► Birthgiver создаёт: Queue[orchestration], Queue[developer], ...
```

---

### Шаг 2 — Context Exploration верхнего цикла

Верхний **Cycle** входит в фазу Context Exploration:
- Читает **Knowledge Base** (пустая на старте)
- Читает **Scenario** (цель, ограничения)
- При необходимости вызывает инструмент `request_decision` → призывает **@human** для уточнений
- Формулирует цели, вехи, ограничения

Все шаги стримятся в **Visibility** (потоковый лог: thinking → output → tool calls).
Каждое решение пишется в **Audit**.

---

### Шаг 3 — Создание первичных тикетов

Верхний **Cycle** вызывает `create_ticket` для каждой крупной подзадачи:
- Тикеты попадают в `incoming` соответствующих **Queue**
- Каждый тикет добавляет узел в **Ticket Graph** со связью `spawned_by=верхний_cycle`
- **Audit** фиксирует `ticket.created` для каждого

**Queue Manager** каждой очереди видит `incoming`:
- Переводит нужные тикеты в `in_queue, priority: N`
- Ненужные → `declined, cause`
- **Audit** фиксирует каждое решение

---

### Шаг 4 — Разрешение зависимостей и готовность тикетов

**Ticket Graph** следит за `blocked_by` связями:
- Когда все зависимости тикета переходят в `done` → тикет становится `ready`
- **Queue** видит `ready` тикет и делает его доступным для **Executor**

---

### Шаг 5 — Исполнение тикета: Executor → Cycle → Model

**Executor** (экземпляр **Role**) забирает `in_queue` тикет с максимальным приоритетом.

Runtime:
1. Выбирает **Model** по **Score System** (лучший балл для данной роли + типа задачи)
2. Создаёт **Cycle** с: промптом **Role**, **Toolset**, делегированным **Budget**, доступом к **Knowledge Base**
3. Тикет переходит в `running` → **Audit**: `ticket.status_changed`

**Cycle** проходит четыре фазы:

**Фаза 1 — Context Exploration:**
- **Model** читает **Knowledge Base** (в рамках scope)
- Читает текущий **Ticket** (цель, checks, evidence предыдущих попыток)
- Читает связанные тикеты из **Ticket Graph** (`related_to`, `evidence_for`)
- Все события → **Visibility** (потоковый лог)

**Фаза 2 — Changes:**
- **Model** вызывает domain-инструменты из **Toolset**: пишет код, запускает команды, редактирует файлы
- Если обнаруживает новую задачу → `create_ticket` → новый **Ticket** в `incoming` нужной **Queue** → **Ticket Graph** получает `spawned_by` связь
- **Budget** обновляется после каждой операции (токены, CPU, файлы)
- **Visibility** стримит каждый tool call в реальном времени

**Фаза 3 — Command Verification:**
- Запускаются команды из `checks` тикета
- Результаты → `evidence` тикета
- **Audit**: `cycle.phase_completed` для фазы verification

**Фаза 4 — Model Verification:**
- **Model** оценивает собранный evidence против критериев тикета
- Если evidence достаточен → тикет → `done` → **Audit**: `ticket.status_changed`
- Если не достаточен → тикет → `needs_review` или следующая попытка
- **Score System** обновляется: баллы модели по (роль, тип задачи, расход бюджета)

---

### Шаг 6 — Провал и перебор моделей

Если **Cycle** завершился неудачей:
- Попытка N+1: **Score System** пессимизирует текущую **Model**
- После 2-3 провалов: runtime выбирает другую/более мощную **Model** по **Score System**
- Тикет остаётся в `running`, меняется только модель
- После исчерпания попыток: тикет → `failed` → **Audit** → **Score System** фиксирует штраф

---

### Шаг 7 — Превышение бюджета

**Budget** достигает лимита:
- Все ongoing **Cycle** получают `budget_exceeded` сигнал → завершают текущую операцию
- Новые **Cycle** не запускаются
- **Audit**: `budget.exceeded`
- Верхний **Cycle** получает `budget_exceeded` как результат
- **Visibility** отображает состояние бюджета в реальном времени

---

### Шаг 8 — Динамическое расширение: Birthgiver

В ходе исполнения какой-то **Executor** призывает роль, которой нет: `@performance_analyst`.

Призыв → `incoming` тикет к **Birthgiver**.

**Birthgiver** через свой **Cycle**:
1. Читает реестр **Role** — роли нет
2. Анализирует задачу — роль нужна
3. Вызывает `create_role` → создаёт **Role** `performance_analyst` с промптом + **Toolset**
4. Создаёт **Queue** `performance` с **Queue Manager** и **Executor**
5. **Audit**: `role.created`, **Visibility** обновляет State View

Исходный тикет теперь может быть перенаправлен в новую очередь.

---

### Шаг 9 — Верификация и завершение Scenario

Все дочерние **Ticket** в `done` → верхний **Cycle** входит в фазу верификации:
- Command Verification: запускаются команды из `verification` блока **Scenario**
- Model Verification: **Model** оценивает суммарный результат против целей **Scenario**

Если проверки пройдены → **Scenario** завершён → **Audit**: финальный `ticket.status_changed`

**Visibility** показывает финальное состояние: все очереди, граф тикетов, расход бюджета, статистику моделей.

---

### Сводная схема флоу

```
Human
  │
  ▼
Scenario ──► верхний Cycle
               │
               ├──► Birthgiver (создаёт Role + Queue)
               │
               ├──► create_ticket ──► Ticket[incoming] ──► Queue
               │                                              │
               │                                    Queue Manager
               │                                     (приоритизирует)
               │                                              │
               │                                         Ticket[in_queue]
               │                                              │
               │                                         Executor
               │                                              │
               │                                    runtime выбирает Model
               │                                    (по Score System)
               │                                              │
               │                                          Cycle
               │                                    ┌─────────────────────┐
               │                                    │  Context Exploration │
               │                                    │  Changes             │──► Budget
               │                                    │  Cmd Verification    │──► Audit
               │                                    │  Model Verification  │──► Visibility
               │                                    └─────────────────────┘
               │                                              │
               │                                     Ticket[done/failed]
               │                                              │
               │                                       Score System
               │                                       (обновляет баллы)
               │
               └──► финальная верификация Scenario
                          │
                        done
```

Audit и Visibility сквозные — присутствуют на каждом шаге.
Budget и Score System — пассивные участники каждого Cycle.
Ticket Graph строится непрерывно по мере создания тикетов.

---

### Статусы тикета и переходы

`incoming` → `blocked` / `ready` → `running` → `done` / `failed` / `needs_review`

## Квест 6: MVP

### Принципы MVP

MVP должен доказать главный инвариант Phase 3: **модель исполняет внутри тикета, runtime управляет флоу, всё наблюдаемо**. Всё остальное — оптимизация.

MVP не делает: распределённый runtime, visual GUI, полный sandboxing, семантический diff, сложный параллельный DAG.

---

### Scope MVP

#### Сущности, которые входят в MVP

| Сущность | MVP-вариант |
|---|---|
| Ticket | Полный lifecycle, JSON-сериализация, все статусы |
| Cycle | Все 4 фазы, строгая последовательность |
| Role | Промпт + фиксированный Toolset, без динамического расширения |
| Queue | Одна очередь `developer`, статическая |
| Queue Manager | Ручная приоритизация (или простой FIFO) |
| Executor | Берёт тикет, запускает Cycle, возвращает результат |
| Model | Одна модель из ollama, без перебора |
| Budget | Счётчик токенов + soft stop, без CPU/RAM |
| Audit | Все trace events в JSON-файл |
| Visibility | Потоковый лог в терминале (не HTML/WS) |
| Ticket Graph | Только `spawned_by` и `blocked_by` |
| Scenario | Plain text + бюджет + одна очередь |

#### Сущности, которые НЕ входят в MVP

| Сущность | Причина отложить |
|---|---|
| Birthgiver | Роли статичны в MVP |
| Score System | Одна модель — нечего сравнивать |
| Knowledge Base | Заменяется передачей контекста в тикете |
| Visibility (HTML/WS) | Достаточно терминального вывода |
| Несколько очередей | Одна очередь покрывает доказательство |
| Параллельные Executor | Один исполнитель за раз |

---

### MVP: шаги реализации (в порядке уровней из Квеста 4)

**Шаг 1 — Audit (уровень 0)**
Реализовать append-only JSON-лог. Функция `emit(event_type, payload)`. Всё пишется в `audit.jsonl`. Нет зависимостей — реализуется первым.

**Шаг 2 — Toolset (уровень 0)**
Реализовать реестр инструментов: базовые (`create_ticket`, `update_ticket`, `read_context`) + domain (bash, file_edit). Инструмент — это callable с именем и scope.

**Шаг 3 — Budget (уровень 1)**
Счётчик токенов. Инициализируется лимитом из Scenario. Метод `consume(tokens)` → возвращает `ok` или `budget_exceeded`. При превышении — emit в Audit.

**Шаг 4 — Ticket (уровень 2)**
Dataclass с полями из спецификации. Методы перехода статусов с валидацией и emit в Audit. JSON-сериализация/десериализация.

**Шаг 5 — Role (уровень 2)**
Dataclass: id, prompt, toolset (subset от глобального). Статически определяется в конфиге (YAML/JSON).

**Шаг 6 — Ticket Graph (уровень 3)**
In-memory граф: dict `{ticket_id: {spawned_by, blocked_by, blocks}}`. Метод `is_ready(ticket_id)` → True если все `blocked_by` в статусе `done`.

**Шаг 7 — Queue (уровень 3)**
Список тикетов + методы: `add(ticket)`, `get_next()` (по приоритету), `update_status(ticket_id, status)`. Вызывает Ticket Graph для проверки `is_ready`.

**Шаг 8 — Executor + Queue Manager (уровень 4)**
Queue Manager: простая логика — переводит `incoming` в `in_queue` с priority=0 (FIFO). Executor: цикл `while queue.get_next(): run_cycle(ticket)`.

**Шаг 9 — Model (уровень 5)**
Обёртка над ollama API: `generate(prompt, tools, context) → stream`. Считает токены, передаёт в Budget.

**Шаг 10 — Cycle (уровень 5)**
Оркестратор четырёх фаз. Каждая фаза — отдельная функция. Фаза Changes: Model вызывает инструменты из Toolset. Фаза Verification: запускает команды из `checks` тикета. Emit в Audit после каждой фазы.

**Шаг 11 — Visibility (уровень 6, MVP-вариант)**
Print-based стриминг в терминал: каждый Audit event → форматированный вывод. Thinking модели выводится по мере генерации. Tool call выводится как `[tool: create_ticket] → {result}`.

**Шаг 12 — Scenario (уровень 7)**
YAML-файл: `goal`, `budget`, `accesses`, `verification`. Загружается при запуске, создаёт верхний Cycle, передаёт Budget.

---

### MVP: минимальный рабочий сценарий

```yaml
# scenario.yaml
goal: "Добавить endpoint GET /health в FastAPI сервис"
budget:
  tokens: 50000
accesses:
  - path: ./src
    mode: read_write
verification:
  commands:
    - "pytest tests/test_health.py"
```

Ожидаемый флоу:
1. Верхний Cycle разбивает цель на тикеты: `dev-1: написать endpoint`, `dev-2: написать тест`
2. `dev-2` заблокирован `dev-1` (`blocked_by: dev-1`)
3. Executor берёт `dev-1`, Model пишет код через `file_edit` tool
4. Command Verification: `pytest` — проходит
5. `dev-1` → `done`, Ticket Graph разблокирует `dev-2`
6. Executor берёт `dev-2`, Model пишет тест
7. Оба тикета `done`, верхний Cycle запускает финальный `pytest`
8. Scenario завершён

Весь процесс виден в терминале как поток событий аудита.
