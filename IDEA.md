# Planfoldr
Deterministic multi-cycle pipeline builder

## Текущие проблемы
- Недетерминизм работы агентов
- Зависимость от качества модели
- Утекающий контекст и забывание на длинной дистанции
- Необходимость верифицировать глазами не только финальный результат, но и сам процесс

## Образ результата
Модель не является центром управления. Модель — один из исполнителей внутри отдельных блоков, где нужно работать с текстом: классифицировать комментарий, сжать лог, предложить patch, извлечь требования, подготовить draft ответа.

Основная иерархия:

Scenario 
- Cycles in cycles
    - Block

Где:
- Scenario — верхнеуровневый контур работы с целью, необходимыми условями, ограничениями;
- Cycle — Зацикленный флоу внутри сценария, например
- Block — атомарная единица действия

## Сценарий
Примеры сценария:
- Исправить падение теста
- Ответить на комментарии в PR
- Принести патч по комментариям в PR
- Поправить flakky-тесты

Внутри каждого сценария должны быть в том или ином виде:
- Цель
- НУ
- Бюджеты
- Циклы

## Цикл
Цикл состоит из набора блоков, разной степени ветвистости.

У цикла обязательно должны быть:
- Способ войти в этот цикл
- Бюджет цикла
- Доступы
- Ограничения
- Вложенные циклы с точками входа и точками обратной связи
- Возможность менять работу вложенных циклов
- Блоки, связи и ветвления между ними

## Части внутри Сценариев, циклов, блоков

### Цель
Цель есть системная (цель всего процесса) и конкретного сценария.

Например у сценария «исправить падение теста» цель единичного сценария — сделать тест зелёным, а цель всего пайплайна — улучшить качество продукта путём увеличения количества покрытых сценариев в продукте. В сценарии важны обе цели.

### НУ (Необходимые условия) достижения цели (?)
Описывает условия, которым решение должно удовлетворять, чтобы цель считалась выполненной; происходит на стадии Verify

### Бюджеты
Бюджеты на запросы, запуски команд (потребление cpu/ram/net/io/...), ...

### Доступы
Набор ограничений, которые 

### Ограничения
Что нельзя делать при достижении цели, по сути является блоком Verify

### Вход
Структурированные поля которые нужны блоку/циклу/сценарию для запуска

### Выход
Структурированные поля которые блок/цикл/сценарий должен отдать выше

выход может запрашивать дополнительную информацию от внешней или от внутреннего цикла

### Контекст
Есть несколько уровней контекстов:
- Текущий у блока
- Глобальный у цикла

## Общий принцип
- constraints go down
- facts go up
- parent can override child
- child can request parent only at allowed points
- context updates are explicit blocks
