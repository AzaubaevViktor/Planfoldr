# Planfoldr

Deterministic multi-cycle pipeline builder.

Planfoldr — это рантайм для описания, запуска, тестирования и инспекции многоцикловых агентных флоу, где модель не является центром управления. Модель — один из исполнителей внутри отдельных задач, а управление процессом остается детерминированным, проверяемым и наблюдаемым.

## Текущие проблемы

- Недетерминизм работы агентов.
- Зависимость результата от качества модели.
- Утекающий контекст и забывание на длинной дистанции.
- Необходимость верифицировать глазами не только финальный результат, но и сам процесс.
- Сложность и дороговизна верификации флоу работы.
- Пропуск части шагов даже моделями последнего поколения.
- Слабая воспроизводимость агентных запусков.
- Сложность тестирования отдельных частей флоу без реальной модели.

## Образ результата

Модель не управляет процессом. Модель вызывается только внутри задач, где действительно нужна работа с текстом или кодом:

- классифицировать комментарий;
- сжать лог;
- предложить patch;
- извлечь требования;
- подготовить draft ответа;
- спланировать структуру проекта;
- проверить результат по описанным критериям.

Детерминированный рантайм управляет:

- жизненным циклом сценария;
- жизненным циклом циклов;
- порядком выполнения задач;
- ветвлениями по типизированным outcome;
- бюджетами;
- доступами;
- контекстом;
- верификацией;
- audit log;
- trace;
- HTML-отчетом.

## Основная иерархия

```text
Scenario
  Cycle
    Task
    Cycle
      Task
```

Где:

- `Scenario` — весь запуск с целью, необходимыми условиями, ограничениями, бюджетами, входами, выходами, циклами и политикой контекста.
- `Cycle` — единица управления флоу внутри сценария. Циклы могут быть вложенными и взаимодействуют друг с другом только в явно описанных точках.
- `Task` — атомарная пользовательская единица работы с типом, задачей, структурированным входом, структурированным выходом и исполнителем.

Старый термин `Block` заменяется на `Task` для новых документов и реализации.

## Терминология

Для реализации используется цепочка:

```text
Intent -> Task -> Execution -> Result
```

- `Intent` — зачем эта работа нужна.
- `Task` — декларативное описание работы в сценарии.
- `Execution` — конкретная попытка выполнения задачи.
- `Result` — структурированный результат выполнения: статус, output, artifacts, budget usage, audit events.

Это нужно, чтобы не смешивать намерение, описание работы и факт исполнения.

## Пример флоу

```text
Event
  -> Scenario
    -> Cycle
      -> Task: Build context
      -> Task: Make changes
      -> Task: Verify changes
      -> Task: Update context
```

Более формально:

```text
Input
  -> Task
  -> Output
  -> Outcome
  -> Transition
  -> Next Task / Next Cycle / Stop / Ask parent / Ask child cycle
```

Ветвление в MVP происходит по enum-статусу результата задачи.

## Сценарий

Примеры сценариев:

- исправить падение теста;
- ответить на комментарии в PR;
- принести patch по комментариям в PR;
- поправить flaky-тесты;
- сгенерировать небольшой проект с нуля и проверить его тестами.

Обязательные поля сценария для MVP:

- `id`;
- `goal`;
- `required_conditions`;
- `constraints`;
- `budgets`;
- `inputs`;
- `outputs`;
- `cycles`;
- `context_policy`.

## Цикл

Цикл состоит из задач, связей между ними и вложенных циклов.

Обязательные поля цикла для MVP:

- `id`;
- `goal`;
- `tasks`;
- `links`;
- `nested_cycles`;
- `budgets`;
- `constraints`.

Циклы разных уровней могут работать независимо и синхронизироваться в явно описанных точках. Родительский цикл может ждать один результат дочернего цикла или все результаты.

## Задача

Обязательные поля задачи для MVP:

- `id`;
- `type`;
- `task`;
- `input_schema`;
- `output_schema`;
- `executor`.

Verifier не является скрытым полем каждой задачи. Проверки результата описываются отдельными verifier-задачами.

Первые типы исполнителей:

- `command`;
- `model`.

Первые типы verifier-задач:

- command verifier;
- schema validation;
- custom script;
- model-based verification.

## Outcome и переходы

MVP-статусы результата:

- `success`;
- `failure`;
- `budget_exceeded`;
- `need_context`;
- `need_decision`;
- `need_answer`;
- `need_inner_cycle`;
- `need_permission`;
- `need_tool`;
- `retry_exceeded`.

Каждый неуспешный outcome должен содержать структурированную причину и evidence, чтобы родительский цикл мог принять решение.

Ошибки не обрабатываются одинаково:

- при `budget_exceeded` родитель решает, можно ли повысить бюджет;
- при `need_permission` родитель решает, выдавать ли дополнительный доступ;
- при невалидном output модели выполняются retry с уточнением схемы;
- после исчерпания retry возвращается `retry_exceeded`.

## Части внутри Scenario, Cycle и Task

### Цель

Цель есть у всей системы и у конкретного сценария.

Например, у сценария "исправить падение теста" локальная цель — сделать тест зеленым. Системная цель — улучшить качество продукта через увеличение количества покрытых сценариев и проверяемость процесса.

### НУ

НУ — необходимые условия достижения цели.

В MVP НУ описываются как цепочка verifier-задач. Цель считается достигнутой, когда verifier-задачи цикла прошли.

### Бюджеты

MVP-бюджеты:

- `max_iterations`;
- `max_tool_calls`;
- `max_model_calls`;
- `max_model_budget`;
- `max_cpu_time`;
- `max_ram`.

Цикл может тратить бюджет сам или делегировать часть бюджета вложенным циклам.

### Доступы

Доступы задаются через allowlist:

- разрешенные tools;
- разрешенные filesystem paths.

Доступы передаются от внешних циклов к внутренним. Внутренний цикл может запросить дополнительные права через `need_permission`.

### Ограничения

Есть два типа ограничений:

- verifiable constraints, которые проверяются verifier-задачами;
- capability constraints, которые определяют, какие инструменты, пути и действия доступны сценарию, циклу или задаче.

### Вход

Структурированные поля, которые нужны сценарию, циклу или задаче для запуска.

### Выход

Структурированные поля, которые задача, цикл или сценарий должны отдать выше.

Выход может содержать запрос дополнительной информации, решения, контекста, инструмента, доступа или запуска внутреннего цикла.

### Контекст

У контекста есть уровни:

- task context and state;
- cycle context and state;
- scenario context and state;
- decision log;
- audit log.

По умолчанию:

- задача может свободно менять свой личный контекст;
- задача может читать явно разрешенный родительский контекст;
- запись в контекст родителя или сценария требует явного права;
- все изменения контекста фиксируются в audit log.

## Общий принцип взаимодействия между циклами

- constraints go down;
- facts go up;
- parent can override child;
- child can request parent only at allowed points;
- context updates are explicit;
- access is explicit;
- budget delegation is explicit.

## Детерминизм и воспроизводимость

Для каждого запуска нужно сохранять:

- полный input задачи;
- полный output задачи;
- запрос к модели;
- ответ модели;
- результат выполнения tool;
- budget before/after;
- prompt id;
- prompt hash;
- переменные prompt-а;
- отрендеренный prompt;
- artifacts.

MVP replay — replay отдельных задач. Full scenario replay можно отложить.

MVP сравнение запусков — сравнение финального статуса. Более детальный diff можно отложить.

## Отчетность

MVP должен производить:

- CLI logs;
- structured trace file;
- static one-page HTML report.

HTML-отчет должен показывать структуру исполнения, давать возможность пошагово смотреть, что делала модель на каждом шаге, и позволять скрывать глубокие уровни вложенности.

## MVP

Первый пользователь: solo-разработчик, который пишет флоу для автоматизированной системы.

Первый интерфейс:

- CLI runner;
- logs;
- static HTML one-page report.

Формат сценариев:

- YAML;
- ссылки на другие YAML;
- ссылки на внешние prompt-файлы.

Первый демонстрационный сценарий:

- e2e-тест приложения с локальной моделью через Ollama;
- модель пишет с нуля простой проект уровня `cli-todo-list` в отдельном git-репозитории;
- результат включает код, тесты и тестовый прогон.

Критерий готовности MVP:

- e2e-тесты с заглушками проходят для успешного сценария;
- e2e-тесты с заглушками проходят для неуспешного сценария;
- e2e с локальной моделью готов пройти;
- runtime производит trace;
- HTML-отчет строится из trace;
- основной принцип доказан: model is not the controller.

## Что не делать в Phase 2

- Не строить production web app.
- Не строить visual scenario builder.
- Не делать distributed runtime.
- Не делать full sandboxing.
- Не делать automatic model routing.
- Не делать full scenario replay.
- Не делать semantic run diff.
- Не делать сложный parallel DAG conflict resolution.

## Flow работы над Planfoldr

- Описать цель и НУ.
- Описать циклы в виде задач и ветвлений.
- Выбрать исполнителей для конкретных задач.
- Описать prompt templates.
- Написать тестовые сценарии для важных частей флоу.
- Запустить unit/e2e с model stubs.
- Запустить optional e2e с локальной моделью.
- Проверить trace и HTML-отчет.
- Инкрементально улучшать сценарий.
